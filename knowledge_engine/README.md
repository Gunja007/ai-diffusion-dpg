# Knowledge Engine DPG

Assembles the complete prompt for each LLM call. Stateless — receives all context in each request.

---

## What this service does

The Knowledge Engine receives a user message (already language-normalised and NLU-tagged by Agent Core) and returns a fully assembled prompt ready for the LLM. It does not call the LLM itself and does not call the Memory Layer — all session history is passed to it in the request body by Agent Core.

Three internal blocks contribute to each prompt:

| Block | Role |
|---|---|
| **Glossary & Domain Vocabulary** | Maps colloquial terms to canonical concepts (e.g. "kaam chahiye" → `market_truth_query`) |
| **Static Knowledge Base** | Semantic RAG retrieval over ingested domain documents (ChromaDB) |
| **Multimodal Input Handler** | Processes images and PDFs embedded in the user message (disabled for PoC) |

## HTTP API

The service runs on port **8001**.

### `POST /assemble_prompt`

Assembles the full system prompt and user message for one LLM call.

**Request body:**
```json
{
  "session_id": "sess-abc123",
  "user_message": "electrician ka kaam kahan milega?",
  "nlu_result": {
    "intent": "market_truth_query",
    "entities": {"trade": "electrician", "location": "Hubli"},
    "confidence": 0.91
  },
  "conversation_history": [
    {"role": "user", "content": "..."},
    {"role": "assistant", "content": "..."}
  ],
  "channel": "cli"
}
```

**Response:**
```json
{
  "system_prompt": "You are Kaam Ki Baat, a labour advisory assistant...\n[glossary context]\n[RAG chunks]",
  "messages": [
    {"role": "user", "content": "electrician ka kaam kahan milega?"},
    ...
  ]
}
```

### `GET /health`

Returns `{"status": "ok"}` when the service is running.

## RAG retrieval

The Static Knowledge Base block uses ChromaDB with `paraphrase-multilingual-MiniLM-L12-v2` embeddings.

| Setting | Value |
|---|---|
| Embedding model | `paraphrase-multilingual-MiniLM-L12-v2` (sentence-transformers) |
| Top-k chunks retrieved | 3 |
| Similarity threshold | 0.65 |
| Vector store | ChromaDB (local, persisted to `data/chroma_db/`) |

Source documents ingested at startup (or via `scripts/ingest.py`):

| Document | Contents |
|---|---|
| `labour_schemes.pdf` | PMKVY, NAPS, other government scheme details including eligibility, benefits, enrolment steps |
| `trade_descriptions.csv` | Trade descriptions, skill requirements per occupation |
| `training_institutes.csv` | Institute names, locations, trades offered |
| `bridge_income.pdf` | Bridge income options, daily wage work, short-term employment |
| `onest_market_truth.csv` | Salary ranges, demand trends by trade and location |

**Retrieval behaviour:** The top-3 most semantically similar chunks are injected into the system prompt. Query phrasing determines which chunks surface — an enrolment-focused query retrieves enrolment steps, not eligibility text, even if both exist in the corpus.

## Ingesting documents

Run once before starting the service (or when source documents change):

```bash
source ../.venv/bin/activate
cd knowledge_engine
python scripts/ingest.py
```

This embeds all documents in `data/` and writes the vector store to `data/chroma_db/`.

## Configuration

| Key | Description |
|---|---|
| `server.port` | HTTP port (default: 8001) |
| `knowledge.embedding.provider` | Embedding backend (`sentence_transformers`) |
| `knowledge.embedding.model` | Model name for embeddings |
| `knowledge.retrieval.top_k` | Number of chunks to retrieve per query |
| `knowledge.retrieval.similarity_threshold` | Minimum cosine similarity to include a chunk |
| `knowledge.sources` | List of document paths to ingest |
| `conversation.max_history_turns` | How many past turns to include in prompt context |
| `conversation.persona` | Persona text injected into every system prompt |
