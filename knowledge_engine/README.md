# Knowledge Engine

Assembles retrieval context for the LLM prompt. Stateless — receives NLU results and session state in every request from Agent Core.

---

## What this service does

The Knowledge Engine receives a user message that has already been language-normalised and NLU-tagged by Agent Core, then returns ranked knowledge chunks for injection into the LLM prompt. It does not call the LLM, does not call the Memory Layer, and does not assemble the final prompt — prompt assembly is Agent Core's responsibility. All session context is passed in the request body.

Three internal blocks run in a fixed order on every request:

| Block | Role |
|---|---|
| **Glossary** | Maps colloquial terms to canonical concepts (config-driven, case-insensitive) |
| **Static Knowledge Base** | Semantic RAG retrieval over ingested domain documents (ChromaDB) |
| **Multimodal Input Handler** | PDF and image extraction — fully implemented but disabled by default |

## HTTP API

The service runs on port **8001**.

### `POST /retrieve`

Run RAG retrieval for one conversation turn. Returns knowledge chunks — not the assembled prompt.

**Request body:**
```json
{
  "session_id": "sess-abc123",
  "user_message": "electrician ka kaam kahan milega?",
  "profile": {},
  "session": {},
  "intent": "market_truth_query",
  "entities": {"trade": "electrician", "location": "Hubli"},
  "sentiment": "neutral",
  "confidence": 0.91,
  "normalised_input": "electrician ka kaam kahan milega",
  "detected_language": "hi"
}
```

`session_id` is required — a missing or empty value returns HTTP 422. All other fields have defaults and are optional.

**Response:**
```json
{
  "session_id": "sess-abc123",
  "chunks": [
    {
      "text": "Hubli mein electrician ki average salary Rs. 15,000–28,000/month...",
      "doc_type": "market_truth",
      "source": "onest_market_truth.csv",
      "always_include": false
    }
  ]
}
```

The endpoint never raises on block failure — partial results are returned. Chunks below the similarity threshold are excluded. Chunks with `always_include: true` bypass the threshold and are fetched separately.

### `GET /health`

Returns `{"status": "ok"}` when the service is running.

## RAG retrieval

The Static Knowledge Base block performs semantic search over the ingested document corpus using ChromaDB.

| Setting | Default |
|---|---|
| Embedding model | `paraphrase-multilingual-MiniLM-L12-v2` (sentence-transformers) |
| Alternative embedding | `text-embedding-3-small` (OpenAI, requires `OPENAI_API_KEY`) |
| Vector store | ChromaDB, local, cosine distance, persisted to `data/chroma_db/` |
| Top-k chunks | 3 (configurable) |
| Similarity threshold | 0.65 (configurable) |

**Filters applied at query time:**

- **Intent filter** — maps the NLU intent to a list of `doc_type` values to search. Only documents of those types are queried. Configured per domain in `intent_filters`.
- **Location filter** — if a district entity is present, matches it against document metadata to prefer geographically relevant chunks.
- **Always-include** — documents with `doc_type: "always_include"` are fetched in a separate query and appended regardless of similarity score.

**Document chunking at ingest time:**

| Source type | Chunking strategy |
|---|---|
| PDF | PyMuPDF text extraction → RecursiveCharacterTextSplitter (chunk_size=400, overlap=50) |
| CSV | One row per chunk |
| TXT / MD | RecursiveCharacterTextSplitter |

## Glossary block

The Glossary block performs case-insensitive substring replacement on `normalised_input` and `entities` before retrieval. Mappings are config-driven — for example:

```yaml
mappings:
  - terms: ["kaam chahiye", "naukri chahiye"]
    canonical: "market_truth_query"
```

Mappings are applied in order. The block can be disabled entirely via `knowledge.blocks.glossary.enabled: false`.

## Configuration

Config is loaded at startup by deep-merging `config/dpg.yaml` (framework defaults) with `config/domain.yaml` (domain values, or the path from `CONFIG_FOLDER` env var). Nothing is hardcoded in source.

| Key | Description |
|---|---|
| `knowledge.blocks.static_knowledge_base.embedding_provider` | `sentence_transformers` or `openai` |
| `knowledge.blocks.static_knowledge_base.embedding_model` | Model name for embeddings |
| `knowledge.blocks.static_knowledge_base.top_k` | Number of chunks returned per query |
| `knowledge.blocks.static_knowledge_base.similarity_threshold` | Minimum cosine similarity to include a chunk |
| `knowledge.blocks.static_knowledge_base.collection_name` | ChromaDB collection name |
| `knowledge.blocks.static_knowledge_base.chroma_persist_dir` | Path to ChromaDB storage directory |
| `knowledge.blocks.static_knowledge_base.sources[]` | List of documents to ingest (path, doc_type, refresh) |
| `knowledge.blocks.static_knowledge_base.intent_filters` | Map of intent → list of doc_types to search |
| `knowledge.blocks.glossary.enabled` | Enable or disable the glossary block |
| `knowledge.blocks.glossary.mappings[]` | List of term → canonical mappings |
| `knowledge.blocks.glossary.apply_to[]` | Fields to apply glossary substitution to |
| `knowledge.blocks.multimodal_input_handler.enabled` | Enable or disable multimodal (default: false) |
| `knowledge.blocks.multimodal_input_handler.image_model` | LLM model for image description when enabled |

## Document ingestion

Run once before starting the service, and again whenever source documents change:

```bash
cd knowledge_engine
uv run python scripts/ingest.py
```

This embeds all configured documents and writes the vector store to `data/chroma_db/`. The service will not return useful chunks if ingestion has not been run.

## Running the service

```bash
cd knowledge_engine
uv run uvicorn main:app --port 8001
```

Set `OPENAI_API_KEY` if `embedding_provider` is set to `openai`.

## Multimodal input handler

The Multimodal Input Handler block is fully implemented but disabled by default (`enabled: false` in config). When enabled:

- **PDF inputs** are extracted via PyMuPDF.
- **Images** are described via an LLM vision call using `HttpLLMWrapper` (routes to Agent Core's LLM proxy).
- **Audio** is not implemented — the block logs "ASR is out of scope" and returns nothing.

No API or interface changes are needed to enable it — set `knowledge.blocks.multimodal_input_handler.enabled: true` in the domain config.

---

## Known gaps

**Audio input not implemented.** The Multimodal Input Handler logs "ASR is out of scope" and returns nothing for audio inputs. ASR/STT is handled upstream by the Reach Layer voice channel, not here.

**No caching layer.** Identical queries (same intent + same entities) re-run full vector search on every call. A caching strategy for high-cardinality repeated queries (e.g. common trade lookups) is an open design question (#18) — no implementation exists yet.
