# Reach Layer — Production Deployment Checklist

This document lists what you need in order to deploy the Reach Layer web UI
(with Google Sign-On enabled) to production. Share this with whoever owns
the prod environment so they can prepare the required values ahead of the
deploy.

---

## 1. Required values to collect before deploy

| # | Name | Type | Who provides it | Notes |
|---|------|------|-----------------|-------|
| 1 | `GOOGLE_CLIENT_ID` | Public string | Google Cloud Console (OAuth 2.0 Client ID) | **Must be created fresh per environment** — CANNOT reuse the one I'm using locally (tied to my email). |
| 2 | `REACH_SESSION_SECRET` | Secret (≥32 random bytes) | Generated by ops at deploy time | Stored as a Kubernetes `Secret`. Never commit to git. |
| 3 | Production hostname | e.g. `https://kkb.example.com` | Ops / DNS owner | Needed to register in the Google OAuth client's **Authorized JavaScript origins**. |
| 4 | TLS certificate / ingress | - | Ops | Production must be served over HTTPS (cookies require `Secure`). |

---

## 2. `GOOGLE_CLIENT_ID` — what it is and how to create it

- A **public** identifier for our app in Google's OAuth system. Safe to put in ConfigMap / env var.
- It is what tells Google "this is the Kaam Ki Baat web app requesting sign-in."
- Currently I'm using a client ID tied to **my personal Google Cloud project / email**. That client ID is only authorized for `http://localhost`, so it will NOT work in production.

**Steps to create a production client ID:**
1. Go to Google Cloud Console → APIs & Services → Credentials.
2. Use a dedicated Google Cloud project for prod (not a personal one).
3. Create Credentials → OAuth client ID → Web application.
4. Under **Authorized JavaScript origins**, add the prod URL, e.g. `https://kkb.example.com`.
5. No redirect URI is needed (we use the ID token / One-Tap flow, not redirect).
6. Configure the **OAuth consent screen** (app name, logo, privacy policy, ToS) before going public.
7. Copy the resulting **Client ID** — that's the `GOOGLE_CLIENT_ID` env value.

Use separate client IDs for dev / staging / prod so a lower environment can't mint prod-looking cookies.

---

## 3. `REACH_SESSION_SECRET` — what it is and why it's needed

- An **HMAC signing key** used by the Reach Layer server to sign the session cookie (JWT, HS256).
- Every time a user successfully signs in with Google, the server mints a cookie (`reach_session`) signed with this secret. On every subsequent request the server verifies the signature to trust the cookie.
- **Without it** (or if it's weak), attackers could forge cookies and impersonate any user.

Rules:
- **Must be ≥32 random bytes** (base64-encoded for easy env var use).
- **Unique per environment** — do not share between dev / staging / prod.
- **Never commit it to git.** In prod it lives in a Kubernetes `Secret` only.
- Rotating it immediately invalidates all active sessions (everyone gets logged out and must sign in again) — this is the expected "kill switch" for compromised sessions.

### Generating it (openssl)

```bash
openssl rand -base64 32
```

`openssl` is a standard Unix tool available on macOS / Linux / most CI runners by default. No install usually needed. The command prints a single random 32-byte string like:

```
Zr8P9n2J+xK0wQhE5tYc7hPqUvM3sRaLdFjNeBkAWXY=
```

That entire string is the value for `REACH_SESSION_SECRET`.

Create the Kubernetes Secret (example):
```bash
kubectl create secret generic reach-layer-auth \
  --from-literal=session-secret="$(openssl rand -base64 32)" \
  -n <namespace>
```

The Helm chart already references this Secret via `secretKeyRef` in
`automation/helm/dpg/reach-layer/templates/deployment.yaml`. The Secret
name / key are configurable via `auth.sessionSecretName` / `auth.sessionSecretKey`
in `values.yaml`.

## 4. Config toggles that MUST change for production

In `dev-kit/configs/<domain>/reach_layer.yaml`:

```yaml
auth:
  enabled: true
  session_ttl_s: 86400        # tune to taste
  cookie_secure: true         # IMPORTANT: must be true in prod (HTTPS only)
```

Locally we override `cookie_secure: false` because we're on plain `http://localhost`. **That override must not be carried into the prod values file** — if `Secure` is false over HTTPS, browsers still accept; but if production is ever accessed over plain HTTP by accident, sessions would leak.

---

## 5. Summary — what to ask prod ops for

1. A **production Google OAuth Client ID** (not tied to a personal account) with our prod hostname in Authorized JavaScript origins → set as `GOOGLE_CLIENT_ID` env var.
2. A **newly generated 32-byte random secret** via `openssl rand -base64 32`, stored in a Kubernetes Secret → injected as `REACH_SESSION_SECRET` env var.
3. **HTTPS ingress** terminating at the prod hostname (required for `Secure` cookies).
4. Confirm `auth.cookie_secure: true` in the prod config.
5. Confirm `.env.local` is NOT shipped in the image (it's already gitignored).

---

## 6. Things to NOT do

- Do not reuse my personal/local `GOOGLE_CLIENT_ID` in prod.
- Do not commit `REACH_SESSION_SECRET` to any repo or YAML.
- Do not deploy with `cookie_secure: false` over HTTPS in prod.
- Do not use the same secret across environments.
- Do not log the session cookie value or the Google ID token credential.
